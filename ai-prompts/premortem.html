02 / METHOD

Premortem frame and assumptions

Imagine the date is 21 August 2026 and the application has caused duplicate cost, an unwanted send, data loss, or local-network exposure. Work backward to the earliest preventable decision.

Skill note. The referenced premortem.md was not present in this repository or in the common local Codex/Claude skill and prompt locations available during review. To avoid blocking, this report uses the standard premortem components: explicit success criteria, assumptions, failure stories, causal chains, evidence, early warnings, prevention/containment, owners, acceptance tests, and go/no-go gates.

Success means

No duplicate or orphaned paid operation; no message sent against current suppression/consent; no private-network research request; no unrecoverable browser-owned data; and deterministic recovery after tab, server, or network failure.

Operating assumptions

Single-user local deployment on 127.0.0.1; imported websites and AI output are hostile; provider calls and sends are real external side effects; browser tabs, network, credentials, and process uptime are unreliable.

Scoring

Likelihood (L), impact (I), and difficulty of detection before harm (D) each score 1–5. RPN = L × I × D. Critical ≥ 70, High 40–69, Medium 20–39, Low < 20.

04 / SCENARIOS

Failure stories, signals, and controls

Each story is phrased as if the loss already occurred, then traced back to code and an acceptance test.

R1 · Critical · 125

“We paid twice because a live provider batch was labeled failed.”

A provider accepted chunk 1. Chunk 2 or the first status poll failed. The UI marked every placeholder and the job failed, stopped resuming it, and the operator retried.

Causal chain

  1. Chunk submissions run sequentially; successful provider IDs exist only in a local array until every chunk completes.
  2. A later failure rejects the entire HTTP request, so already-created provider jobs are not returned to the browser.
  3. After a successful response, the same browser catch handles both submission and monitoring errors and marks all results failed.
  4. Reload recovery resumes only the newest job whose status is still running/stopping; a failed or second active job is ignored.
  5. A missing runtime credential after server restart triggers the same false-failure path.

Evidence

src/ai/providerBatchService.js:292–321, 399–424, 519–550, 685–741 · public/app.js:2205–2231, 2430–2443 · server.js:10–18

Early warnings

  • Local job is failed while provider dashboard still shows validating/in-progress.
  • Repeated deterministic customId values across separate provider batch IDs.
  • Provider cost/usage increases without matching completed local results.
  • Log has “submitted” followed by a poll/network/credential error and no terminal provider status.

Prevent / contain

  • Persist an operation and each provider ID immediately around every external side effect.
  • Separate submission state from monitoring state; poll errors become monitoring-degraded, never terminal.
  • Return and persist partial chunk receipts; reconcile ambiguous operations before allowing retry.
  • Resume all active provider operations with bounded backoff.

Acceptance test

Inject failure after provider acceptance, after chunk 1 of 2, during first poll, after server restart, and after response loss. Exactly one remote batch per chunk exists; local state remains reconcilable; retry never submits until reconciliation proves absence.

R2 · Critical · 80

“Two tabs sent the same scope and then overwrote each other’s state.”

Both tabs loaded the same IndexedDB snapshot. Neither knew the other was processing. Both initiated paid work; one tab’s final writes hid the other operation.

Causal chain

  • state.processing is in-memory and tab-local.
  • The application has no navigator.locks, BroadcastChannel, shared lease, revision check, or idempotent start operation.
  • Job/result IDs are newly generated per invocation, so provider and Resend idempotency do not collapse duplicate tab operations.
  • IndexedDB writes use unconditional put; edits and settings are last-write-wins.

Evidence

public/app.js:2265–2304, 3074–3103 · public/modules/storage.js:53–111 · repository-wide search: no navigator.locks or BroadcastChannel usage

Early warnings

  • Two running jobs for the same project/record/template digest.
  • Duplicate recipients with different result IDs.
  • Settings or edited HTML revert after switching tabs.

Prevent / contain

  • Acquire a named browser-wide lease for processing, restore, and sending.
  • Broadcast active operation state and render other tabs read-only for that scope.
  • Add record revisions and compare-and-swap writes for editable objects.
  • Use server operation IDs as the ultimate duplicate barrier.

Acceptance test

Open two Playwright contexts against the same persistent profile and click Process/Send simultaneously. Only one external call occurs; the losing tab shows the owner and can observe or take over only after lease expiry and reconciliation.

R3 · Critical · 75

“A prospect website reached an internal service and crashed the research host.”

The attacker’s DNS answered publicly during validation and privately when Chromium connected. The page then loaded active JavaScript in a browser launched without its sandbox and consumed unbounded resources before post-render size checks ran.

Causal chain

  • Node validates DNS, but Chromium resolves and connects separately; the route handler caches the validation promise by origin.
  • The browser starts with --no-sandbox, JavaScript enabled, ignored TLS errors, and broad public-origin subresources.
  • The document byte check runs after page.content(); script/XHR totals are not bounded during transfer.
  • Research may launch a new browser for the main page and up to three contact pages per record.

Evidence

src/research/websiteFetcher.js:65–84, 91–105, 119–142 · src/research/researchService.js:88–120

Early warnings

  • Final socket IP differs from validated DNS answers.
  • Chromium CPU/RAM spikes or child processes survive timeouts.
  • Research accesses unexpected origins or large script/XHR transfers.

Prevent / contain

  • Pin DNS to the validated public address at connect time through a controlled proxy/resolver; validate every redirect and socket.
  • Restore Chromium sandboxing and strict TLS; fail closed if unavailable.
  • Apply streaming per-response, per-page, and per-job byte/time budgets before allocation.
  • Disable unneeded browser features and pool isolated contexts with forced teardown.

Acceptance test

A DNS-rebinding fixture, redirect-to-private fixture, IPv4-mapped IPv6 target, oversized stream, endless JavaScript, service worker, websocket, and download all fail closed without reaching the private listener or exceeding the resource budget.

R4 · Critical · 100

“We sent after suppression, lost partial receipts, and could not prove the confirmed content.”

The confirmation dialog stayed open while suppression state changed. Send reused its cached preflight snapshot; a later Resend batch failed after an earlier one succeeded, so the endpoint returned no receipts and the browser stored none.

Causal chain

  • Suppressions are read once before the dialog and cached in state.resendPreflight.
  • Final Send reuses the cached items/suppressions rather than opening a fresh IndexedDB transaction.
  • Server sends chunks sequentially but throws away earlier success receipts if a later chunk fails.
  • Idempotency hashes only sorted item IDs and From address, not message/recipient digests or a distinct send operation.
  • Bulk postal address is checked for non-empty input but not guaranteed inside the canonical message; test-recipient mode borrows another record’s consent evidence.

Evidence

public/app.js:3009–3049, 3074–3103 · src/resend/resendGateway.js:87–124, 135–205 · src/routes/resendRoutes.js:51–84

Early warnings

  • Provider shows sent messages absent from deliveryHistory.
  • Preflight age/revision is unavailable or confirmation stays open for minutes.
  • Same idempotency key maps to changed recipient/content.
  • Final HTML lacks the configured postal address or recipient-specific unsubscribe target.

Prevent / contain

  • Bind confirmation to a short-lived digest of exact messages, sender, consent evidence, and suppression revision.
  • Re-read all eligibility state atomically immediately before send.
  • Persist per-chunk operation and receipts; return partial status instead of losing successes.
  • Use operation/chunk/message digests for idempotency and explicitly model an intentional resend.

Acceptance test

Add a suppression after preflight, change a recipient/content after confirmation, drop the response after the first successful chunk, restart, and retry. No newly suppressed address is contacted; exact receipts recover; no duplicate occurs; a deliberate resend requires a new reviewed operation.

R5 · High · 48

“A large import/export froze the UI and exhausted the gateway.”

The documented 1,000-record limit is not enforced by the provider-batch route, whose schema permits 50,000 records. Provider helpers call response.text() without applying providerResponseBytes. Browser backup and delivery export use synchronous zipSync; OPFS staging occurs only after the full archive exists in memory.

src/routes/gatewayRoutes.js:28–37 · src/ai/gatewayProvider.js:31–61 · src/ai/providerBatchService.js:49–79 · public/app.js:2992–2998 · public/modules/backup.js:37–63

Controls / acceptance

  • Enforce config-derived counts, nesting, fields, bytes, and concurrency at every API boundary.
  • Stream and abort provider bodies at the byte cap.
  • Move compression to a worker/stream directly to OPFS and report progress/cancel.
  • Load-test maximum supported records and responses under explicit RAM, event-loop-lag, and wall-clock budgets.
R6 · High · 48

“A backup passed superficial checks, then crashed or corrupted restored state.”

unzipSync expands the complete archive before expanded bytes are counted, so the decompression-bomb limit is post-allocation. Store payloads are checked only for array shape, depth, and prototype keys; there are no per-store schemas, record count limits, key uniqueness checks, or relationship validation before one-transaction commit.

public/modules/backup.js:5–6, 67–105, 108–122

Controls / acceptance

  • Stream entries with compressed/expanded, entry-count, per-entry, ratio, and time limits enforced before allocation.
  • Validate each store with versioned schemas and cross-store referential integrity.
  • Show a dry-run plan and reject unknown required categories/duplicate keys.
  • Fuzz archives and prove zero mutation on every failure.
R7 · High · 45

“The user completed paid work in temporary mode and lost it on close.”

Any IndexedDB open failure falls back to an in-memory repository. A status warning is shown, but processing, research, sending, restore, and editing remain enabled; later status updates can displace the warning. Browser persistence is optional and there is no automatic recoverable journal outside IndexedDB.

public/modules/storage.js:142–212, 245–250 · public/app.js:4173–4203

Controls / acceptance

  • Use a persistent, non-dismissible degraded-mode banner and explicit acknowledgement.
  • Disable paid/external side effects by default in temporary mode.
  • Offer immediate encrypted/manual backup and periodic checkpoint prompts.
  • Simulate VersionError, blocked upgrade, quota exhaustion, eviction, and close/reopen.
R8 · Medium · 36

“An old API surface bypassed the new gateway contract.”

Modern /gateway/* mutations call requireGatewayRequest, while legacy record, job, result, regeneration, export, and model-sync mutations do not. Many also lack strict request schemas or config-derived maxima. Origin checks reduce drive-by browser risk but absent-Origin clients are accepted, and the server does not enforce the expected loopback Host header.

src/routes/dataRoutes.js:32–148 · src/routes/processingRoutes.js:77–149 · src/routes/resultRoutes.js:62–238 · src/routes/modelCatalogRoutes.js:32–43 · src/middleware/security.js:53–72

Controls / acceptance

  • Remove unused legacy routes or put all unsafe methods behind one global token/origin/host policy.
  • Keep the signed webhook as an explicit exception.
  • Apply strict schemas, body limits, and expensive-route concurrency/rate limits.
  • Test missing/spoofed Origin, Host, CSRF, duplicate headers, content types, and public-bind misconfiguration.
R9 · High · 48

“Shutdown cut across a side effect and left no reliable audit trail.”

Provider-batch HTTP helpers have no AbortSignal timeout. Shutdown clears credentials immediately, waits for HTTP connections, and force-exits after eight seconds; the force path does not close SQLite or the rotating logger. There is no idempotent shutdown guard or shared abort controller for in-flight work.

src/ai/providerBatchService.js:49–79 · server.js:10–29 · src/utils/logger.js:45–72

Controls / acceptance

  • Add deadlines to every outbound call and a global shutdown AbortController.
  • Stop accepting new work, checkpoint operation state, abort safely, then flush logs and close DB.
  • Make shutdown idempotent and report unresolved operations on next start.
  • Signal during upload, submit, poll, research, send, log rotation, and SQLite write in tests.
R10 · Medium · 24

“We shipped because the assertions were green while the release command was red.”

All 158 Vitest assertions passed, but npm test exited 1 with an unhandled fork-runner startup timeout and worker termination timeouts. Formatting also fails in four files. Existing provider-batch E2E exercises happy-path reload only, while the dominant failures require network/crash/partial-effect and multi-context tests.

Controls / acceptance

  • CI requires command exit code 0, not just assertion counts.
  • Stabilize worker configuration and capture open-handle diagnostics.
  • Run formatting check in pre-commit/CI and fix the four reported files.
  • Add deterministic fault injection for every critical scenario.
05 / PLAN

Sequenced risk-reduction plan

Order is based on irreversible external harm first, then hostile boundaries, then recoverability and throughput.

Now · containment
  • Default provider batch to Standard API or mock; require an “experimental / reconcile in provider dashboard” acknowledgement.
  • Disable live bulk send and hostile-site research outside an explicitly controlled evaluation profile.
  • Open only one tab; export a backup before paid work; keep provider dashboards visible during tests.
  • Fix formatting drift and preserve a clean checkpoint/branch before structural changes.
Days 1–3 · P0
  • Implement durable external-operation ledger, partial chunk receipts, nonterminal poll failures, and resume-all reconciliation.
  • Add browser-wide operation lease and same-scope idempotency.
  • Bind Resend confirmation to fresh eligibility state and persist partial sends.
Days 4–7 · boundary
  • Ship DNS-pinned, sandboxed, resource-budgeted research or keep research disabled.
  • Apply global unsafe-method protection, strict schemas, loopback Host enforcement, response limits, and request deadlines.
Week 2 · data
  • Replace synchronous archive expansion/compression with bounded streaming/worker flows.
  • Add versioned backup schemas, referential validation, dry-run restore, and temporary-mode guardrails.
Release · proof
  • Run crash/network/two-tab fault matrix and verify provider-dashboard reconciliation.
  • Require lint, format, Vitest, Chromium/Firefox/WebKit E2E, secret scan, audit, and clean-worktree checks to exit zero.
  • Document residual risks, rollback, telemetry signals, and on-call/runbook ownership.
06 / GATES

Go / no-go criteria

These are observable release conditions, not “best effort” aspirations.

NO-GO if any is true

  • A remote operation can exist without a durable local operation ID and reconciliation path.
  • Two tabs can create two provider/send calls for one logical operation.
  • A poll/network/credential error marks a nonterminal provider job failed.
  • Final send does not revalidate current suppression/consent and exact content.
  • Research can run unsandboxed, resolve separately from validation, or exceed streaming budgets.
  • Backup expands before limits or commits records without schemas.
  • Any required quality command exits nonzero.

GO when all are true

  • Fault injection proves no duplicate/orphan operation across response loss, restart, and partial chunks.
  • Multi-context E2E proves one active owner and safe takeover.
  • Resend receipts and eligibility are durable, fresh, digest-bound, and auditable.
  • Private-network and hostile-page research fixtures fail closed within resource budgets.
  • Maximum supported import/export/restore completes within published RAM/time budgets.
  • A clean clone passes every required command twice consecutively.

Prompt 1 — Durable provider-batch operations, recovery, and multi-tab exclusion
Solves R1 + R2 provider-generation paths + R9 ambiguity 
You are a senior reliability engineer working in the AI Batch Personalizer repository. Implement an end-to-end, crash-recoverable external-operation protocol for native provider batches, and prevent concurrent browser tabs from starting the same logical operation.

Context and observed defects
- The browser is intended to own projects, records, jobs, and results. The loopback server owns credentials and external provider calls.
- src/ai/providerBatchService.js submits chunks sequentially and does not durably persist each provider batch ID before attempting the next chunk. A later error rejects the whole request and can orphan earlier provider jobs.
- public/app.js uses one catch block around submission plus monitoring; any polling, network, or missing-credential error after successful submission marks all placeholder results and the job failed.
- reload recovery resumes only the newest provider-batch job in running/stopping state.
- tab-local state.processing is the only active-operation guard. Two tabs can start distinct jobs with distinct IDs for the same project/record/template scope.
- Server restart clears runtime credentials; this must pause monitoring for credential re-entry, not relabel remote work terminal.

Required design
1. Introduce an explicit operation state machine. At minimum distinguish: preparing, submitting, partially-submitted, submitted, monitoring, monitoring-degraded, awaiting-credential, cancel-requested, reconciling, completed, partially-completed, canceled, failed-terminal, and ambiguous. Document legal transitions and make illegal transitions fail loudly in development/tests.
2. Generate a stable operationId in the browser before the first gateway call. Persist it atomically with the job, exact record IDs, provider/model, template digest, prompt-input digest, execution settings, creation timestamp, and state. Every submit/status/cancel request must carry operationId.
3. Add a small durable server-side operation ledger in SQLite for external-effect metadata only (no credentials and no full email bodies). Include operationId unique, requestHash, operation type, provider/model, chunk ordinal, provider batch/file IDs, provider-visible reconciliation key/name/metadata, state, last error class, attempts, and timestamps. Add a migration and repository with transactions.
4. Make submit idempotent: same operationId + same requestHash returns/reconciles the existing operation; same operationId + different hash returns 409. Persist a pre-call intent before each remote create and persist the provider receipt immediately after each successful create before attempting the next chunk.
5. Use provider-supported idempotency primitives where available. Where a create endpoint has no idempotency primitive, put operationId and chunk ordinal into provider-visible metadata/name/file identity and implement a provider-specific lookup/reconciliation path. Never automatically resubmit an ambiguous chunk merely because a response was lost.
6. Return partial submission state and all durable successful chunk receipts even when a later chunk fails. Do not turn a partial external success into a generic thrown error that hides receipts. The browser must atomically merge receipts and display an actionable reconciliation state.
7. Separate submit failures from monitor failures. A status poll error, timeout, 429/5xx, missing credential, server restart, or temporary provider parse error is nonterminal. Preserve running placeholders, use capped exponential backoff with jitter, display the cause and next retry, and resume after credential restoration. Only a verified terminal provider status can terminally fail unresolved results.
8. Resume and reconcile every active provider operation, not just one. Bound concurrent polling and avoid thundering herds. An operation that cannot be reconciled must remain visibly ambiguous with provider IDs and instructions; block a new same-scope submit until the user explicitly resolves it.
9. Add a browser-wide lease using navigator.locks when available plus BroadcastChannel for status propagation. Provide an IndexedDB lease fallback with owner tab ID, operation ID, heartbeat, expiry, and compare-and-swap revision. Processing, provider-batch monitoring/cancel, restore, and retry must respect the lease. A non-owner tab is read-only for the active scope and can observe progress. Takeover requires expiry followed by provider reconciliation.
10. Add optimistic revisions to job/result writes touched by recovery so a stale tab cannot overwrite a newer terminal state.
11. Add deadlines and AbortSignals to every provider batch HTTP request. Integrate a global shutdown AbortController: stop new submissions, checkpoint state, finish or safely abort bounded work, flush logs, and close SQLite. Shutdown must be idempotent.

Compatibility and safety constraints
- Preserve the existing browser-owned data model for ordinary content. The server ledger stores only the minimum external-effect journal needed to guarantee recovery.
- Do not persist credentials, authorization headers, full prompts, research content, or generated email bodies in the server ledger or logs.
- Preserve current provider adapters and public API behavior where safe; version changed payloads explicitly.
- Never claim exactly-once delivery when a provider lacks the necessary primitive. Expose “at-most-once after reconciliation” or “ambiguous” honestly.
- Keep Standard API behavior functional, but route its same-scope duplicate guard through the browser lease.

Tests to add before considering the change complete
- Unit-test every legal/illegal state transition and idempotency hash conflict.
- Fault-inject after pre-call ledger write, after provider accepts a chunk but before receipt write, after receipt write but before HTTP response, after chunk 1 of 2, during first poll, while reading output, during cancel, and during shutdown.
- Restart the server at every fault point and prove the operation is discovered/reconciled without an automatic duplicate.
- Remove and restore credentials during monitoring; state must become awaiting-credential then resume.
- Return 429 with delta-seconds and HTTP-date Retry-After, repeated 5xx, malformed JSON, and a hanging body; verify capped backoff/deadlines.
- Open two Playwright contexts with the same persistent browser profile and race Process, Retry, and Cancel. Assert one submit call, one operationId, visible ownership in both tabs, and safe takeover after simulated owner death.
- Create two active operations and reload; both must resume under bounded polling.
- Assert logs and the ledger contain no credential or full-content material.

Acceptance criteria
- For every injected failure, the number of provider-created chunks equals the intended unique chunk count and never increases merely because the client retries.
- Local state never says terminal failed while a known provider chunk is nonterminal.
- Every known remote provider ID is discoverable from the operation ledger and surfaced in diagnostics.
- A same-scope second tab cannot create external work.
- All new and existing lint, format, unit, integration, security, and provider-batch E2E commands exit zero.

Deliverables
- Code and migrations, state-machine documentation, a short recovery runbook, provider-specific reconciliation notes, tests, and an updated architecture diagram showing the browser lease and minimal server operation ledger.
- In your final response, list changed files, schema/API changes, fault cases proven, commands run with exit codes, and any provider limitation that still leaves an ambiguity window.
Prompt 2 — Research browser SSRF, sandbox, and resource-budget hardening
Solves R3 and the research portion of R5/R9 
Act as a security engineer specializing in SSRF and hostile-browser isolation. Redesign the research fetch path so imported, redirected, and subresource URLs are treated as adversarial and cannot access private networks, disable containment, or consume unbounded resources.

Observed evidence
- src/research/websiteFetcher.js resolves a hostname with Node, then lets Chromium independently resolve/connect. The route handler caches the validation promise by origin, creating a DNS-rebinding/time-of-check-time-of-use gap.
- Chromium is launched with --no-sandbox, JavaScript enabled, ignoreHTTPSErrors true, and broad public-origin script/XHR access.
- responseBytes is enforced only after page.content(), after the browser has downloaded/rendered the document. Total subresource bytes, sockets, CPU, and memory are not bounded.
- collectResearch may launch one browser for the primary page and one for each of up to three contact pages per record. Provider batch preparation can repeat this across many records.
- BatchSubmitSchema permits 50,000 records although configured import support is 1,000.

Required changes
1. Write a concise threat model covering direct IPs, DNS rebinding, CNAME/redirect chains, IPv4-mapped IPv6, link-local metadata, localhost aliases, wildcard DNS, proxy environment variables, malicious TLS, service workers, websocket/WebRTC, downloads, data/blob/file schemes, browser exploits, and decompression/resource exhaustion.
2. Replace separate validate-then-browser-resolve behavior with connect-time enforcement. Preferred architecture: route all Chromium HTTP(S) traffic through a local controlled proxy/resolver that resolves each hostname, rejects any answer in blocked ranges, pins the chosen public address for that connection while preserving Host/SNI certificate validation, revalidates every redirect/new socket, and records the actual remote IP. If you choose a different architecture, prove it closes the same TOCTOU gap.
3. Reject a host if any DNS answer is blocked; normalize and classify IPv4, IPv6, IPv4-mapped IPv6, decimal/octal/hex and Unicode hostname edge cases. Bound DNS answers, CNAME depth, lookup time, redirect count, and origin count. Do not trust system proxy settings unless explicitly validated.
4. Remove --no-sandbox. Use the platform browser sandbox and fail closed with a clear configuration error if safe containment cannot start. Set ignoreHTTPSErrors false. Use a dedicated ephemeral browser user-data directory/context, no stored credentials, no permissions, no downloads, no clipboard, no geolocation, no camera/microphone, no WebRTC local-IP discovery, no extensions, and blocked service workers.
5. Minimize active content. If research quality permits, disable JavaScript by default and make rendered-JS mode an explicit bounded fallback. In rendered mode, block websocket/eventsource, downloads, navigation outside policy, and unnecessary resource types; limit scripts/XHR to justified same-site origins or a small bounded set. Treat browser output as tainted regardless.
6. Enforce limits while streaming, before allocation: compressed and decompressed bytes per response, total bytes per page, total bytes per record, number of requests/origins/redirects, wall time, idle time, DOM size, extracted text, and concurrent contexts. Abort immediately at the limit. Never call response.text() or page.content() on an unbounded body.
7. Replace launch-per-page behavior with a bounded browser pool only if isolation remains strong: a fresh context per page/record, strict teardown in finally, process health checks, maximum pages per process, and kill/recycle on timeout or budget violation. No orphan process may survive a job.
8. Enforce context.config.limits.records at the gateway batch API and a much smaller explicit research record/page concurrency budget. Validate record field count, field bytes, nesting depth, website length, and research depth with shared schemas.
9. Add structured, redacted security telemetry: requested host, normalized host, resolved/connected IP classification, redirects, blocked reason, bytes, request count, duration, and teardown outcome. Never log query secrets, page bodies, prompts, credentials, or full email content.
10. Update SECURITY.md, PERFORMANCE.md, .env.example, and the UI so disabled/degraded research is explicit and does not imply completion.

Required adversarial tests
- A deterministic DNS resolver fixture returns public on validation and private on connection; verify the private listener sees zero requests.
- Cover 127/8, 0/8, RFC1918, link-local metadata, multicast, reserved, unique-local IPv6, IPv4-mapped IPv6, localhost with trailing dot/case, alternate IP encodings, CNAME to private, mixed public/private answers, and redirect public→private→public.
- Test a valid public page whose script fetches a private URL, opens websocket/EventSource, registers a service worker, triggers a download, creates a blob/file navigation, and attempts WebRTC discovery. All disallowed actions must be blocked.
