## Required Test Taxonomy

Create or update a test strategy and traceability matrix covering all 30 categories below. These categories overlap because some describe test levels, some describe techniques, and others describe quality characteristics. Do not duplicate identical tests merely to create 30 folders. A well-designed test may satisfy multiple categories when the mapping is accurate and documented.

Each category must have:

- A stable category identifier.
- One or more requirements or risks.
- Named test cases with stable IDs.
- The target component or workflow.
- Test level and technique.
- Automation status.
- Framework and command.
- Fixture or environment requirements.
- Expected result.
- CI execution cadence.
- Traceability to the relevant source, requirement, defect, or README claim.
- Actual execution evidence when the test can run in the available environment.

The required categories are:

1. **Static Testing** — source linting, formatting, type checking, dependency analysis, configuration validation, documentation linting, and static security analysis.
2. **Component (Unit) Testing** — isolated functions, classes, modules, validators, parsers, formatters, template processors, and state transitions.
3. **Component Integration Testing** — interactions among internal modules such as input parsing, template processing, provider adapters, output generation, logging, and persistence.
4. **System Testing** — the complete application operating as a unified system in a controlled environment.
5. **System Integration Testing** — integration with real architectural boundaries such as files, browsers, APIs, storage, subprocesses, or databases, using safe test equivalents.
6. **End-to-End Testing** — realistic user workflows from installation or launch through input, personalization, results, export, and error recovery.
7. **Acceptance Testing** — verifiable business outcomes and user-oriented acceptance criteria.
8. **Functional Testing** — confirmation that documented features behave according to requirements.
9. **Non-Functional Testing** — reliability, maintainability, observability, security, accessibility, compatibility, usability, and performance characteristics.
10. **Black-Box Testing** — testing through public interfaces without relying on implementation details.
11. **White-Box Testing** — branches, conditions, internal state transitions, exception paths, and complex implementation logic.
12. **Exploratory Testing** — documented charters, scenarios, observations, risks, and reproducible findings.
13. **Smoke Testing** — fast validation that the application installs, launches, and performs its essential workflow.
14. **Regression Testing** — permanent protection for existing behavior and previously resolved defects.
15. **Confirmation Testing** — focused retesting demonstrating that each corrected defect is actually fixed.
16. **API Testing** — request/response validation, schemas, status codes, authentication behavior, rate limits, retries, timeouts, malformed responses, and compatibility when APIs exist.
17. **Database Testing** — schema, migrations, constraints, transactions, persistence, corruption handling, and concurrency when a database exists.
18. **Installation and Upgrade Testing** — clean installation, reinstall, dependency installation, configuration migration, upgrade compatibility, and documented quick-start commands.
19. **Compatibility Testing** — supported runtime versions, browsers, operating systems, file encodings, package versions, and input formats.
20. **Internationalization and Localization Testing** — Unicode, emoji, accented characters, non-Latin scripts, right-to-left text, locale-sensitive formatting, and encoding boundaries.
21. **Accessibility Testing** — keyboard access, focus behavior, labels, landmarks, contrast, semantic structure, screen-reader compatibility, and WCAG 2.2 AA expectations when a web UI exists.
22. **Usability Testing** — clarity of workflows, validation messages, error recovery, discoverability, and prevention of destructive mistakes.
23. **Performance Testing** — latency, throughput, memory consumption, CPU consumption, startup time, and processing efficiency.
24. **Load Testing** — expected concurrent use and realistic batch sizes.
25. **Stress Testing** — behavior beyond expected limits, including saturation, throttling, and controlled failure.
26. **Endurance (Soak) Testing** — sustained execution for detecting memory leaks, resource leaks, unbounded logs, or degradation.
27. **Scalability Testing** — behavior as records, templates, providers, files, or concurrent jobs increase.
28. **Recovery Testing** — interruption, restart, provider outage, partial writes, corrupt input, unavailable storage, and recovery from partial batches.
29. **Security Testing** — injection, unsafe rendering, path traversal, secret leakage, insecure storage, dependency risk, log injection, malformed input, and authorization boundaries.
30. **Penetration Testing** — authorized, non-destructive testing against local or disposable environments only.

If a category is genuinely inapplicable—for example, database testing when the application has no database—do not invent a database merely to satisfy the taxonomy. Instead:

- Add a verifiable applicability case demonstrating that the layer does not exist.
- Explain the evidence.
- Map the underlying risk to the nearest real storage or integration boundary.
- Mark the category as “Not Applicable—Verified,” not silently omitted.

## Test-Case Design Requirements

Use appropriate standard test-design techniques throughout the suite:

- Boundary-value analysis.
- Equivalence partitioning.
- Decision tables.
- State-transition testing.
- Pairwise or combinatorial testing.
- Property-based testing where it adds value.
- Fuzz testing for parsers, templates, URLs, filenames, and structured input.
- Mutation testing for critical validation or transformation logic where practical.
- Contract testing for provider adapters and external interfaces.
- Golden-file or snapshot testing only when the output is stable and the snapshot is meaningfully reviewed.

Add edge cases applicable to the real application, including:

- Null, undefined, blank, missing, zero-length, and whitespace-only values.
- Empty, single-record, normal, large, and maximum-supported batches.
- Malformed JSON, CSV, text, configuration, templates, and provider responses.
- Missing, extra, reordered, duplicated, or differently cased fields and headers.
- UTF-8 BOMs, alternate line endings, delimiter conflicts, quoting, embedded commas, and multiline values.
- Unicode, emoji, accented characters, non-Latin text, and right-to-left text.
- Extremely long values and oversized files.
- Missing, repeated, escaped, adjacent, or unresolved template placeholders.
- Duplicate records and conflicting identifiers.
- Empty, truncated, invalid, delayed, and partially successful AI-provider responses.
- Authentication failures, permission failures, timeouts, offline conditions, rate limits, throttling, and transient server errors.
- File-permission failures, invalid paths, interrupted writes, unavailable storage, and simulated disk-write failure.
- Cancellation, restart, retry, concurrent processing, ordering, and idempotency.
- Partial-batch failure without losing successfully completed records.
- Log-write failure without crashing the primary workflow.
- Untrusted content, prompt injection, HTML/script injection, formula injection, path traversal, and log injection, as applicable.

Do not add tests for features that the application does not have.

## Coverage Expansion and Measurement

Interpret “40%–80% more coverage” rigorously and transparently.

1. Establish the baseline before modifying tests.
2. Report baseline and final:
   - Statements.
   - Lines.
   - Functions.
   - Branches.
   - Critical-module coverage.
   - Number of meaningful test objectives mapped to each category.

3. For executable source coverage, target a **40%–80% reduction in uncovered executable lines and branches**, calculated as:

   `improvement = (uncovered_before - uncovered_after) / uncovered_before × 100`

4. For each test category with existing coverage, target a 40%–80% relative increase in meaningful covered risks, requirements, branches, or workflows—not merely a larger file count or assertion count.

5. For a category with no baseline tests, create a risk-based initial suite containing, where applicable:
   - A successful case.
   - A negative or boundary case.
   - A failure-recovery or observability case.

6. If the requested improvement is mathematically impossible because coverage is already high, code is generated, or a path requires unavailable external infrastructure:
   - State the exact limitation.
   - Exclude only demonstrably non-executable, generated, vendor, fixture, or test-support code.
   - Achieve the maximum meaningful improvement.
   - Never fabricate coverage figures or write trivial tests to inflate metrics.

7. Configure CI coverage thresholds at least as strict as existing thresholds and high enough to prevent regression from the new stable baseline.

8. Consider per-file or per-module thresholds for critical code such as:
   - Input parsing.
   - Template substitution.
   - Provider adapters.
   - Output generation.
   - Batch orchestration.
   - Retry and recovery behavior.
   - Secret redaction.
   - Logging.
   - Error handling.

## Graceful Error Handling and Recovery

Implement or improve centralized error handling without hiding defects.

The application must:

- Validate inputs early and return actionable messages.
- Distinguish user/configuration errors, provider errors, transient errors, security errors, and internal defects.
- Preserve the original cause for diagnostics while showing safe user-facing messages.
- Retry only transient failures.
- Use bounded retries with exponential backoff and jitter where appropriate.
- Honor provider retry guidance such as `Retry-After`.
- Avoid retrying invalid credentials, malformed requests, or permanent validation errors.
- Continue processing independent records when one record fails, when safe.
- Preserve completed work during partial batch failures.
- Support cancellation and graceful shutdown where the architecture permits.
- Avoid partial or corrupt output through atomic writes or equivalent safe persistence.
- Return meaningful process exit codes or visible UI status.
- Provide a final failure summary and a clear recovery action.
- Fall back gracefully when optional functionality such as file logging is unavailable.
- Never convert an unexpected programmer error into silent success.

Add confirmation and regression tests for every defect fixed during this refactor.
